## Q1: 如何用工程手段建立基于大模型的评测标准？

**提问:**

> 对于这种混合型架构，常规的接口跑通是远远不够的，你需要建立基于大模型的评估标准:
> 1. 路由准确率兜底: `supervisor_agent.py` 能不能在遇到极其模糊的意图时，100% 准确地把需要查分数的请求分发给 `match_agent.py`？
> 2. 防幻觉穿透测试: 需要针对大量 Edge Cases 设计测试计划，确保 `risk_assessor.py` 这种硬风控逻辑绝对不能失效。

**解答:**

这个问题的核心在于区分"什么可以 100% 确定性测试"和"什么需要 LLM 辅助评测"。

**Layer 1 — 确定性 fallback 路由 (100% 覆盖)**

`_fallback_route` 是纯 Python 函数，不依赖任何 LLM。它的逻辑是硬编码的关键词匹配与优先级判断：画像缺失 → `profile_agent`，搜索关键词 → `web_search_agent`，就业关键词 → `career_agent`，分数/位次 → `match_agent`，默认 → `synthesis_agent`。

评测方案: 创建 `data/eval/routing_golden.json`，30 条黄金用例覆盖 7 类意图。`tests/test_supervisor_routing.py` 中的 `test_all_golden_cases` 逐条遍历，验证 fallback 路由与预期一致。30 条全部通过，实现 100% 确定性覆盖。

**Layer 2 — LLM 路由评测框架**

当 API 可用时（`DEEPSEEK_API_KEY` 已加载），使用 `temperature=0` 评测 LLM 路由决策。筛选完整画像的 case，逐条调用 supervisor agent 并比对路由结果。底线要求 70% 准确率。

**Layer 3 — 技能边缘案例与防穿透**

创建 `data/eval/skills_edge_cases.json`，80+ 条精心构建的边界值与对抗输入：

- `risk_assessor`: 20 个 case，覆盖四大天坑 × 所有 tier 组合、医学预算临界值（79999 vs 80000）、None 输入、空字符串、SQL 注入
- `reality_checker`: 11 个 case，覆盖 tolerance 边界（-9 vs -8）、高分低报警告（35 vs 36）、零 tolerance、负数输入
- `decision_heuristics`: 11 个 case，覆盖空画像/全画像/部分画像/None 输入
- `roi_calculator`: 6 个 case，覆盖零薪资/负薪资/极大值

## Q2: 如何用工程手段强制约束 Synthesis 节点，使其不篡改底层硬规则的严重程度？

**提问：**

> 大模型天生具有"端水"和"和稀泥"的倾向。如果 `reality_checker.py` 已经给出了"绝对劝退"的强硬信号，终点节点在生成自然语言时，极有可能会把它软化成"建议您再慎重考虑"。如何用工程手段强制约束？

**解答:**

这是 Agentic 系统中最隐蔽也最危险的失控模式——确定性风控信号在自然语言生成层被 LLM 稀释。纯 Prompt Engineering 不可靠。

**SynthesisGuard 三层防线 (纯 Python，零 LLM 调用)**:

**Layer 1 — 信号检测 (`detect_signals`)**

从 `GraphState` 中提取所有硬规则信号，构建 `RiskSignal` 对象：
- `risk_assessment.is_risk=="true"` → critical 信号
- `reality_check.is_realistic=="false"` → critical 信号

**Layer 2 — Prompt 注入 (`build_guard_prompt`)**

当检测到 critical 信号，向 System Prompt 前置注入 5 条强制规则：
- 第一段必须是风控警告
- `must_say` 必须逐字出现
- 禁止转折弱化词
- 高风险必须用确定性词汇

**Layer 3 — 输出校验与强制修正 (`validate_output` + `enforce`)**

LLM 输出后执行三项硬检查：
1. `must_say` 是否逐字出现在输出中
2. 风险警告是否在前 500 字符内
3. 是否包含端水关键词

任一检查失败 → 强制在输出前插入风控拦截块。

## Q3: 多轮对话的状态继承与覆盖

**提问:**

> `profile_agent.py` 负责多轮对话收集画像，但在真实报考咨询中，用户经常"反悔"。LangGraph 如何处理这种状态变量的随时覆盖与回溯？

**解答:**

**Layer 1 — Checkpoint 持久化**

LangGraph 内置 Checkpoint 机制：每次图执行完，完整 state 被序列化保存。下一轮调用时只需带上相同的 `thread_id`，图引擎自动恢复 state。

提供两种后端:
- `MemorySaver`: 内存存储，适合开发/测试
- `SqliteSaver`: SQLite 持久化，生产环境重启不丢失

**Layer 2 — Profile Merge 策略**

每次调用 profile_agent 时：
1. 从当前 query 抽取新字段
2. 从 state 中读取已有画像
3. Merge: 新值覆盖旧值
4. Diff: 检测哪些字段发生了覆盖，生成 `ProfileChange` 记录
5. 将变更追加到 `profile_history`

**Layer 3 — 回溯与审计**

当用户连续变更时（北京 → 上海 → 深圳），每次覆盖都会记录完整的变更轨迹，可通过 API 查询。

## Q4: 工具异常的重试策略颗粒度

**提问:**

> 对于 `sql_tools.py` 这种关键节点，如果用户输入的学校名字带有错别字导致查不到任何数据，是直接抛错给用户，还是在 Tool 内部建立一套类似模糊搜索或重试的防御机制？

**解答:**

原 `sql_tools.py` 只有两种返回状态：成功或失败。任何细微偏差都会导致硬截断。系统需要的是**在工具内部完成多级降级**。

**解决方案: `ToolResult` 统一返回值 + SQLTools 四级降级链**

| 等级 | 含义 | 触发条件 |
|------|------|---------|
| `exact` | 精确命中 | 省名标准化后的精确匹配 |
| `fuzzy` | 模糊匹配 | 专业名 LIKE `%keyword%` |
| `degraded` | 降级查询 | 年份放宽、多策略回退 |
| `empty` | 无结果+建议 | 探测数据库中可用数据并告知 |
| `error` | 查询异常 | 数据库连接/语法错误 |

**四级降级链**：
- L1: 省份标准化后精确匹配（"粤"→"广东省"）
- L2: 专业名 LIKE 模糊匹配
- L3: 放宽年份限制
- L4: 探测数据库中可用组合，生成建议列表

## Q5: CRM 用户画像与断点续传

**提问:**

> 如果把"对话记忆"升级成"用户资产管理"，在 `data/sql_schema/` 下加一张 CRM 表，用户哪怕过了一个月再来问，系统能否直接从 CRM 加载历史画像？

**解答:**

**CRM 数据库表设计**：以手机号为主键，结构化存储省份、选科、分数、目标城市、预算等字段。

**工作流程**：
1. 用户访问 → `CRM.load_profile(phone_number)`
2. 如果画像存在: 注入 initial state
3. LangGraph 执行完整对话流程
4. synthesis 结束 → `CRM.save_profile(phone_number, user_profile)`
5. 下次访问: 直接从 CRM 恢复画像

**商业价值**：
- 断点续传：用户一月后回来，CRM 直读画像
- 资产沉淀：结构化客户标签可支撑精准服务
- 会话统计：`session_count` + 时间戳提供活跃度分析

## Q6: Supervisor 路由关键词混合测试 — Prompt Fuzzing

**提问:**

> 能否编写自动化测试脚本，像搭积木一样动态拼装出大量的边缘测试用例？

**解答:**

**解决方案: `tests/test_supervisor_fuzzing.py` (21 项测试，720+ 生成用例)**

**测试维度 (6 类)**:

| 维度 | 用例数 | 示例 |
|------|--------|------|
| 标准路由测试 | 504 | 12 意图 × 6 实体噪声 × 7 情绪噪声组合 |
| 噪音干扰测试 | 5×9 | 情绪前缀/后缀不改变核心路由决策 |
| 错别字鲁棒性 | 7 | "西按郊通大学"、"记算机专叶" |
| 混合意图边界 | 5 | 注入混合/多关键词交叉/对抗注入 |
| 关键词优先级链 | 6 | 搜一下 > 就业 > 分 > 默认 |
| CRM 集成验证 | 4 | 断点续传端到端验证 |

**验证策略**：不校验 LLM 生成了什么内容，只校验路由是否指对了方向。路由对了，底盘就稳了。

## Q7: 现代化可视化界面重设计

**提问:**

> 现有的 Flask UI 过于粗糙和老土，能否做成封装化、可视化、动态化、易操作的现代化界面？

**解答:**

将原有的 132 行纯文本调试界面完全重写为专业级可视化前端。

| 维度 | 旧版 | 新版 |
|------|------|------|
| 布局 | 单列卡片堆叠 | 双栏布局（主聊区 + 右侧面板） |
| 对话 | `<pre>` 纯文本 | Chat Bubble 气泡 + 打字动画 + 头像 |

| 画像 | 无 | 右侧面板实时展示用户画像 |
| 视觉 | 黑白灰三色 | 深蓝+金色渐变（教育平台风格） |
| 响应式 | 无 | 移动端自适应 |

## Q8: Vue 3 现代化前端架构升级

**提问:**

> 现有的 Flask UI 仍然是单文件HTML实现，缺乏组件化、模块化和可维护性。能否引入现代前端框架？

**解答:**

**架构升级: 从单文件HTML到Vue 3组件化架构**

**技术选型:**

| 技术 | 版本 | 用途 |
|------|------|------|
| Vue 3 | 3.3.4 | 前端框架，响应式数据绑定 |

|------|------------|------------|
